# 风神录魔a：更复杂的子机

魔a的子机系统则完全脱离了传统的 `slist` 式的设计。大体上说，在高速时，子机会沿着自机走过的路径移动；在低速时，子机需要与自机保持相对的静止。

为了实现高速时的子机机制，我们一般会创建一个数组（表）来记录自机最近若干帧的历史位置。假设我们需要知道自机在 n 帧前的位置，我们有一个长为 n+1 的表 `list`，在自机的 frame 回调中该表作这样的更新：

```lua
for i = 1, n - 1 do
    list[i] = list[i + 1]
end
list[n] = {self.x, self.y}
```

那么更新之后我们就可以用 `list[1]` 来获取 n 帧前的自机位置。

（或者，如果 `list` 的长是 n，那也可以在更新之前用 `list[1]` 获取 n 帧前的自机位置，更新之后的 `list[1]` 则是 n-1 帧前的位置）

那么我们大体的写法就有了：高速时（如果自机发生移动）我们就对 `list` 作上述的更新从而刷新自机历史位置，低速时就让 `list` 里的位置与自机作同步的移动（这可以通过读取 `dx,dy` 属性来实现）。

那么[附录](../mainline/appendix)有具体的代码，我们看一下里面有哪些实现上的细节。

在 init 回调，定义了 `self.xlist, self.ylist` 两个表来存放自机的历史位置。`self.delay = 10` 表示相邻子机之间的间隔为 10 帧，4 个子机一共需要 40 帧之间的历史位置。而 frame 回调先更新历史位置再更新子机，根据前面的讨论，`xlist,ylist` 的长度应为 41。

init 回调还定义了 `self.spx, self.spy` 两个表来存储子机的实际位置，定义了 `self.valid` 表来存储子机的有效性（0~1）。

frame 回调中更新自机历史位置和更新子机的代码拆分成了两个函数。

## `UpdateSupports` 方法

子机的数据就是位置 `spx[i], spy[i]` 和有效性 `valid[i]`。

有效性 `valid[i]` 由火力档位的平滑过渡 `self.support` 计算。例如，假设 `support` 为 3.2（在 3P 和 4P 之间过渡，更接近 3P），那么第 3 个子机的有效性为 $3.2-3+1=1.2 \ge 1$，`valid[3]` 记为 1；第 4 个子机的有效性为 $3.2-4+1=0.2$，`valid[4]` 记为 0.2。

通过读取 `xlist, ylist` 获取子机对应的目标位置，第 i 个子机对应 i*delay 帧之前的自机位置，于是可以计算对应的索引值。`#self.xlist` 是 lua 的语法，功能是获取表 `self.xlist` 的数组长度。

获取目标位置后用一个经典的线性插值，让 `spx[i], spy[i]` 逼近目标位置。

## `UpdateHistory` 方法

更新自机历史记录也就是修改 `xlist, ylist` 表。

希望还原魔a的一个特性：当火力提升时，新出现的子机与前一个子机重叠。这里采取的策略是将无效的子机（也就是，火力提高时可能新出现的子机）对应的历史位置全部抹除，统一替换为最后一个有效子机的位置。

于是计算了一个 `m` 来判断哪些位置要抹除历史。你可能注意到计算 `m` 时使用的取整是向上取整的 `math.ceil`，而不是向下取整的 `int` 或 `math.floor`，这是有理由的。假设火力等级由 4P 降到 3P，如果用向下取整，那么 `m` 使用的火力等级（`support` 取整）在过渡阶段就会变化，导致子机在完全消失之前，对应的历史位置就会抹除，造成我们不想要的效果。

`m` 所在的位置是有效的子机，不抹除，而小于 `m` 的位置是无效的子机，需要抹除。于是不论是在高速的更新历史位置，还是低速的同步移动，对 `xlist,ylist` 的更新都只在 `m` 到 `n` 之间的位置进行。对于 1 到 `m-1` 的位置，则强制让它们与 `m` 位置同步。

你可能还注意到，它在高速和低速检测自机移动时使用了不同的属性。低速下为了获取自机的移动，使用了 `dx,dy` 属性，这在任意 obj 中表示前后两帧位置的变化量，在其他的教程中也有所介绍。而高速下为了获取自机的移动，却使用了一组神秘的 `__move_dx, __move_dy` 属性。实际上这组属性是在自机 data 中设置的，详见[frame.move事件](../dataer/player-system#frame-control)。从命名上，下划线开头，尤其是双下划线开头的变量/属性，是希望自己是私有的、不被其他地方使用的，我也十分建议读者最好不要使用这类变量/属性。

那么为什么这里特意使用了这组属性？这是考虑到自机位置可能被某些弹幕强行修改：阿空的地底太阳会让自机强制移动，也有一些特殊弹幕让自机强制静止。

总之，这里设置为在高速时使用 `__move_dx,__move_dy`，只考虑按下方向键的情况而不管自机实际上怎样移动；在低速时使用 `dx,dy`，只考虑实际的位移，而不管玩家怎么按方向键。

但还是要强调一下，我们在制作弹幕、自机或者别的东西时，考虑到代码规范，最好不要使用这类下划线开头的 “私有” 变量。
